BIG PICTURE: WHAT THIS COURSE IS REALLY ABOUT
============================================================
DevOps means taking code from a developer's machine and making it run reliably on a server. In this exam, the usual journey is:

1. Linux commands: create files/folders, check system info, manage permissions.
2. Flask API: create backend routes that return JSON.
3. Git/GitHub: save code history and push code to a remote repository.
4. Networking/runtime: confirm app is reachable on a port and keep app running.
5. Docker: package Flask app so it runs the same everywhere.
6. CI/CD: automatically test and build app using GitHub Actions.
7. AWS EC2: deploy Dockerized Flask app on a cloud server and test using public IP.

If an examiner asks "what happens when your API is opened from internet?", answer:
A client sends an HTTP request to the EC2 public IP on port 5000. AWS Security Group allows port 5000. Docker maps EC2 host port 5000 to container port 5000. Flask receives the request, matches the route, executes the Python function, returns JSON, and sends an HTTP status code like 200, 201, 400, or 404.

WEEK 2 — PERMISSIONS, USERS, PROCESSES (DESCRIPTIVE)
============================================================
WHAT THIS WEEK IS ABOUT
Linux protects files using permissions. DevOps engineers must know permissions because SSH keys, scripts, logs, and deployment files need correct access.

PERMISSION FORMAT EXAMPLE
-rwxr-xr--
- first character: file type (- file, d directory)
- next three: owner permissions
- next three: group permissions
- last three: others permissions

PERMISSION NUMBERS
r = read = 4
w = write = 2
x = execute = 1

Common numbers:
755 -> owner can read/write/execute; group/others can read/execute. Good for scripts.
644 -> owner can read/write; group/others can read. Good for normal files.
600 -> owner can read/write only. Good for private config.
400 -> owner can read only. Required for SSH .pem key.

PERMISSION COMMANDS
chmod 755 run.sh
# Makes script executable by everyone, writable only by owner.

chmod 644 public.txt
# Normal readable text file permission.

chmod 600 private.txt
# Private file, only owner can read/write.

chmod 400 final.pem
# SSH key permission. SSH may reject key if permissions are too open.

USER/GROUP COMMANDS
whoami
# Shows current username.

id
# Shows user ID, group ID, and groups.

groups
# Shows your groups.

sudo groupadd devteam
# Creates group.

sudo usermod -aG devteam $USER
# Adds current user to group. -aG means append group; do not forget -a.

PROCESS COMMANDS
ps aux
# Shows all running processes. Use when finding Flask/Python process.

ps aux | grep app.py
# Finds Flask app process.

kill PID
# Gracefully stops process.

kill -9 PID
# Force kills process. Use only if normal kill fails.

python3 app.py &
# Runs app in background, but may stop when terminal closes.

jobs
# Shows background jobs in current terminal.

SYSTEM RESOURCE COMMANDS
df -h
# Disk usage. If disk is full, apps may fail.

free -h
# Memory usage.

uptime
# Shows how long server has been running and load average.

WHAT THIS WEEK IS ABOUT
A Bash script is a file containing terminal commands. DevOps uses scripts to automate repeated tasks like setup, deployment, and system reporting.

IMPORTANT SCRIPT RULES
- Start with #!/bin/bash. This tells Linux to run file using Bash.
- Use chmod +x script.sh to make it executable.
- Run using ./script.sh.
- Variables cannot have spaces around =.
- Use $variable to read variable.
- Use $(command) to store command output.
- Use $((math)) for arithmetic.

WEEK 5 — FLASK CRUD API (DESCRIPTIVE)
============================================================
WHAT THIS WEEK IS ABOUT
Flask is a Python web framework. In this lab, Flask creates a REST API. A REST API exposes endpoints/URLs such as /api/students. It returns JSON and uses HTTP methods.

WHY jsonify IS USED
jsonify converts Python dictionaries/lists into JSON responses and sets the correct Content-Type header.

WHY request.get_json() IS USED
request.get_json() reads JSON sent by curl or a frontend in POST/PUT requests.

FLASK CRUD APP.PY TEMPLATE
cat > app.py <<'EOF'
from flask import Flask, jsonify, request
from flask_cors import CORS

app = Flask(__name__)
CORS(app)

students = [
    {'id': 1, 'name': 'Ali', 'grade': 'A'},
    {'id': 2, 'name': 'Sara', 'grade': 'B'}
]
next_id = 3
VALID_GRADES = {'A', 'B', 'C', 'D', 'F'}

def find_student(sid):
    return next((s for s in students if s['id'] == sid), None)

@app.route('/')
def home():
    return 'Flask API is running. Try /api/health or /api/students', 200

@app.route('/api/health')
def health():
    return jsonify({'status': 'ok', 'message': 'Flask is running'}), 200

@app.route('/api/students', methods=['GET'])
def get_students():
    return jsonify(students), 200

@app.route('/api/students/<int:sid>', methods=['GET'])
def get_student(sid):
    student = find_student(sid)
    if not student:
        return jsonify({'error': 'Not found'}), 404
    return jsonify(student), 200

@app.route('/api/students', methods=['POST'])
def add_student():
    global next_id
    data = request.get_json()
    if not data or 'name' not in data or 'grade' not in data:
        return jsonify({'error': 'name and grade are required'}), 400
    grade = data['grade'].upper()
    if grade not in VALID_GRADES:
        return jsonify({'error': 'Invalid grade'}), 400
    new_student = {'id': next_id, 'name': data['name'], 'grade': grade}
    students.append(new_student)
    next_id += 1
    return jsonify(new_student), 201

@app.route('/api/students/<int:sid>', methods=['PUT'])
def update_student(sid):
    student = find_student(sid)
    if not student:
        return jsonify({'error': 'Not found'}), 404
    data = request.get_json()
    if not data:
        return jsonify({'error': 'JSON body required'}), 400
    if 'name' in data:
        student['name'] = data['name']
    if 'grade' in data:
        grade = data['grade'].upper()
        if grade not in VALID_GRADES:
            return jsonify({'error': 'Invalid grade'}), 400
        student['grade'] = grade
    return jsonify(student), 200

@app.route('/api/students/<int:sid>', methods=['DELETE'])
def delete_student(sid):
    global students
    student = find_student(sid)
    if not student:
        return jsonify({'error': 'Not found'}), 404
    students = [s for s in students if s['id'] != sid]
    return jsonify({'message': 'Deleted successfully'}), 200

@app.route('/api/students/count')
def count_students():
    return jsonify({'count': len(students)}), 200

@app.route('/api/students/search')
def search_students():
    grade = request.args.get('grade')
    name = request.args.get('name')
    results = students
    if grade:
        results = [s for s in results if s['grade'].lower() == grade.lower()]
    if name:
        results = [s for s in results if name.lower() in s['name'].lower()]
    return jsonify(results), 200

if __name__ == '__main__':
    app.run(host='0.0.0.0', port=5000, debug=False)
EOF

RUN FLASK
pip3 install flask flask-cors
python3 app.py

1.1 IP Addresses and Interfaces  
Every computer connected to a network has at least one IP address, a unique numerical label that 
identifies it on the network. Think of it as a postal address: without it,  no one knows where to send 
data.  
 
Linux systems have network interfaces, named connections to networks. Common interface names 
are eth0 (wired ethernet), wlan0 (Wi -Fi), and lo (loopback, the machine talking to itself). The 
loopback address 127.0.0.1 alway s refers to the local machine and is also accessible as localhost.  
 
# View all network interfaces and their IP addresses  
$ ip addr show  
 
# Shorter, cleaner output  
$ ip addr show | grep 'inet '  
 
# Older alternative (still widely used)  
$ ifconfig  
 
In the out put, look for a line like inet 192.168.1.5/24. The number before the / is your IP address. The 
/24 is the subnet mask, it defines which part of the address identifies the network vs the individual 
host.  
 
1.2 Testing Connectivity with ping  
ping sends small test packets to another machine and measures how long the reply takes. It is the 
first tool you reach for when diagnosing a network problem.  
# Ping Google's DNS server (tests internet connectivity)  
$ ping 8.8.8.8  
 
# Ping by hostname (also tests DNS resolut ion)  
$ ping google.com  
 
# Send only 4 packets then stop  
$ ping -c 4 google.com  
 
# Ping the loopback, tests the local network stack  
$ ping 127.0.0.1  
 
If ping 8.8.8.8 works but ping google.com fails, the network is fine but DNS resolution is broken. This 
distinction immediately narrows down where the problem lies, a critical debugging skill.

 Method 3: Using systemd (production standard)  
systemd is the Linux service manager. It can start applications automatically at boot, restart them if 
they crash, and manage their logs. This is how real production services are managed.  
# Create a service file for your  Flask app  
$ sudo nano /etc/systemd/system/flaskapp.service  
 
[Unit]  
Description=Samantha Flask App  
After=network.target  
 
[Service]  
User=samantha  
WorkingDirectory=/home/samantha/devops/week5  
ExecStart=/usr/bin/python3 app.py  
Restart=always  
 
[Install]  
WantedBy=multi -user.target  
 
# Reload systemd, enable and start the service  
$ sudo systemctl daemon -reload  
$ sudo systemctl enable flaskapp  
$ sudo systemctl start flaskapp  
 
# Check its status  
$ sudo systemctl status flaskapp  
 
# Stop and disable  
$ sudo systemctl s top flaskapp  
$ sudo systemctl disable flaskapp  
 
Restart=always in the service file means systemd will automatically restart your app if it crashes. This is 
the minimum you need for any production service, it ensures your app recovers from unexpected errors  
without manual intervention.  
 
Logs                                                                                                     . 
3.1 Why Logs Matter in DevOps  
Logs are the single most important tool for understanding what is happening on a runnin g server. 
When an application crashes at 2am, when a user reports that a page is not loading, when a 
deployment breaks silently, logs are where you find the answer. A DevOps engineer who cannot 
read logs cannot debug production systems.  
 
Every application writes log messages as it runs. These messages record events: a request was 
received, a database query ran, an error occurred, a user logged in. By reading the logs, you 
reconstruct exactly what happened and when.

What is Containerisation?                                                                 . 
1.1 The Problem Containers Solve  
You have heard the phrase 'it works on my machine'. This is one of the oldest and most frustrating 
problems in software development. Let suppose, Samantha writes a Flask app on her laptop running 
Ubuntu 22. It works perfectly. She sends it to a teammate whose machine runs a different Python 
version and different library versions. The app crashes immediately.  
 
The traditional solut ion was a Virtual Machine (VM),  a complete simulated computer running inside 
your computer, with its own OS, libraries, and application. VMs solve the consistency problem but 
are heavy: each VM needs gigabytes of disk space and minutes to start, because it runs a full 
operating system.  
 
Containers are a lighter alternative. A container packages your application together wi th everything it 
needs to run  the correct Python version, the exact library ver sions, the configuration files  into a single 
portable unit. Unlike a VM, a container does not include a full OS. It shares the host machine's OS 
kernel, making it start in seconds and use far less disk space.  
1.2 Containers vs Virtual Machines Feature  Virtual Machine  Container  Includes OS  Yes — full OS per VM (GBs) No — shares host kernel (MBs) Startup time Minutes  Seconds  Isolation  Full hardware-level isolation  Process-level isolation  Portability  Large, harder to move  Small, runs anywhere Docker is installed  Use case  Running different OSes, legacy systems  Microservices, modern app deployment  
 
1.3 Key Docker Concepts Concept What It Is  Image  A read-only blueprint for a container — like a class in OOP. Built from a Dockerfile  Container  A running instance of an image — like an object created from a class  Dockerfile  A text file with step-by-step instructions for building a Docker image Docker Hub  A public registry where Docker images are stored and shared (like GitHub but for images) Docker Compose  A tool for defining and running multi-container applications using a single YAML file  Volume  A way to persist data outside the container so it survives container restarts

Writing a Dockerfile                                                                             . 
4.1 What is a Dockerfile? 
A Dockerfile is a plain text file that contains step-by-step instructions for building a Docker image. 
Each instruction creates a new layer in the image. Docker caches each layer, so if you change only 
your app code, Docker only rebuilds the layers that changed making subsequent builds very fast.  
 
4.2 Dockerfile for the Flask CRUD API 
Create a Dockerfile inside your Week 5 Flask project directory:  
 
# Dockerfile  
 
# Step 1: Start from an official Python base image  
FROM python:3.11-slim  
 
# Step 2: Set the working directory inside the container  
WORKDIR /app  
 
# Step 3: Copy the requirements file first (for layer caching)  
COPY requirements.txt .  
 
# Step 4: Install Python dependencies  
RUN pip install --no-cache-dir -r requirements.txt  
 
# Step 5: Copy the rest of the application code  
COPY . . 
 
# Step 6: Tell Docker which port the app listens on (documentation only)  
EXPOSE 5000  
 
# Step 7: The command to run when the container starts  
CMD ["python3", "app.py"]

Docker Compose                                                                                . 
5.1 What is Docker Compose and Why Use It?  
Real applications rarely consist of a single container. A typical setup might include a backend API 
container, a database container, and a frontend container  all needing to communicate with each 
other. Starting, configuring, and linking multiple containers with separate docker run commands is 
complex and error-prone.  
 
Docker Compose solves this by letting you define your entire multi-container application in a single 
file called docker-compose.yml. With one command docker compose up  all containers start together, 
correctly configured and networked. This makes local development and testing dramatically simpler.  
 
5.2 The docker-compose.yml File  
Create a new project directory for a multi-container application:  
$ mkdir -p ~/devops/week9/fullstack && cd ~/devops/week9/fullstack  
 
Create the docker-compose.yml file:  
# docker-compose.yml  
version: '3.8'  
 
services:  
 
  # Service 1: Flask backend API  
  api: 
    build: ./api              # Build image from ./api/Dockerfile  
    container_name: flask-api  
    ports:  
      - '5000:5000'           # host:container port mapping  
    environment:  
      - FLASK_ENV=development  
    volumes:  
      - ./api:/app            # Mount code for live reload during dev  
    networks:  
- app-network
 
  # Service 2: Simple web server for the frontend  
  frontend:  
    image: nginx:alpine       # Use official nginx image directly  
    container_name: frontend  
    ports:  
      - '8080:80'             # Serve frontend on host port 8080  
    volumes:  
      - ./frontend:/usr/share/nginx/html  # Mount our HTML files

What is CI/CD?                                                                                    . 
1.1 The Problem CI/CD Solves  
Before CI/CD, software teams integrated and deployed code manually. Developers worked in 
isolation for days or weeks, then merged everything together only to discover dozens of conflicts and 
broken features. Deployments were manual, error -prone, and stressful events that sometimes took 
entire weekends.  
 
CI/CD automates the entire process from c ode push to deployment. The moment a user pushes a 
commit to GitHub, an automated pipeline springs into action running tests, building the application, 
and verifying everything works without any human intervention.  
 
1.2 Continuous Integration (CI)  
Continuo us Integration means integrating code changes frequently ideally every time a developer 
pushes to the repository and automatically verifying that the change does not break anything. A CI 
pipeline typically:  
• Checks out the latest code from the repository  
• Installs all dependencies  
• Runs automated tests to verify correctness  
• Builds the application (e.g. builds a Docker image)  
• Reports success or failure back to the developer immediately  
 
1.3 Continuous Delivery (CD)  
Continuous Delivery extends CI by automaticall y preparing the application for deployment after 
every successful build. The code is always in a deployable state. In a full CD pipeline, the application 
might also be automatically deployed to a staging or production server. In this lab we focus on CI 
and the build stage of CD.

Pipeline  The full automated sequence of steps that runs on every code push  
Workflow  A GitHub Actions pipeline: defined in a YAML file inside your repository  
Job A group of steps that run together on th e same machine (runner)  
Step A single action inside a job, run a command, or use a pre -built action  
Runner  The virtual machine that GitHub provides to execute your workflow  
Trigger  The event that starts the workflow e.g. a push to main, or a pull reques t 
Secret  An encrypted variable stored in GitHub for sensitive values like passwords 
and API keys

GitHub Actions Fundamentals                                                         . 
2.1 How GitHub Actions Works  
GitHub Actions is GitHub's built -in CI/CD platform. It is free for public repositories and has a 
generous free tier for private ones. Workflows are defined as YAML files stored inside your 
repos itory at a specific location: .github/workflows/. GitHub reads these files automatically and runs 
them when the trigger event occurs.  
 
Every workflow file follows the same structure: a name, one or more trigger events, and one or more 
jobs. Each job contai ns a list of steps that run sequentially on a GitHub -hosted runner (a fresh 
Ubuntu, Windows, or macOS virtual machine).

Writing Automated Tests for Flask                                                    . 
3.1 Why Tests are Essential in CI  
A CI pipeline is only as useful as the tests it runs. Without tests , the pipeline just checks that the 
code can be installed not that it actually works correctly. Automated tests are the safety net that 
catches bugs before they reach users.  
 
Python's pytest framework makes writing and running tests simple. Flask provides a built -in test 
client that lets you make HTTP requests to your app in tests without actually starting a server.

4.5 Using Secrets for Sensitive Values  
Never put passwords, API keys, or tokens directly i n your workflow YAML file, it is public and anyone

      - name: Log in to Docker Hub

known secret values in logs, but any transformation of the secret (e.g. base64 encoding) will not be

 Cloud Computing Fundamentals                                                       . 
1.1 What is Cloud Computing? 
Before the cloud, companies that wanted to run a server had to buy physical hardware, find a place 
to put it, connect it to the internet, cool it, power it, and maintain it. This was expensive, slow, and 
required specialist staff. If demand suddenly doubled, there was no quick way to add capacity.  
 
Cloud computing means renting computing resources  servers, storage, networking, databases from 
a provider over the internet, on demand, and paying only for what you use. Instead of buying a 
server, you click a button and have a virtual server running in minutes. When you no longer need it, 
you delete it and stop paying.  
 
1.2 Cloud Service Models Model  Full Name  What You Get  Example IaaS Infrastructure as a Service  Raw virtual machines, storage, networking  you manage the OS and everything above  AWS EC2, Google Compute Engine PaaS  Platform as a Service  A managed platform you deploy code, the provider manages the OS and runtime  AWS Elastic Beanstalk, Heroku  SaaS  Software as a Service  A fully managed application you just use it  Gmail, GitHub, Slack   
In this lab you will use IaaS specifically AWS EC2 where you get a raw virtual machine and are 
responsible for everything that runs on it. This is the most fundamental cloud model and the one 
DevOps engineers work with most directly.  
 
1.3 What is AWS and EC2? 
AWS (Amazon Web Services) is the world's largest cloud provider, used by companies from startups 
to governments. EC2 (Elastic Compute Cloud) is AWS's virtual machine service. Each virtual 
machine is called an instance. You choose the operating system (Ubuntu, Amazon Linux, Windows), 
the hardware specs (CPU, RAM), and the region (which data centre in the world to use).  
 
1.4 Key AWS and EC2 Terms Term  Meaning Instance  A single virtual machine running in AWS  AMI Amazon Machine Image , the OS template used to launch an instance (e.g. Ubuntu 22.04) Instance TypeThe hardware specs of the instance , e.g. t2.micro (1 vCPU, 1GB RAM, free tier) Security Group A virtual firewall that controls which ports are open to the internet  Key Pair  An SSH key pair , the private key (.pem file) lets you log in to your instance securely

Lab Task 1: Live Cloud Deployment  
Deploy your Flask CRUD API as a Docker container on an AWS EC2 instance. The application 
must be publicly accessible via the EC2 public IP address when the instructor tests it. 
1. Launch a t2.micro Ubuntu 22.04 EC2 instance named user-devops-server with a key pair 
called devops-key 
2. Configure the Security Group to allow inbound TCP on port 22 (SSH) and port 5000 (Flask 
API) 
3. SSH into the instance from your local terminal: ssh -i ~/.ssh/devops-key.pem 
ubuntu@YOUR_PUBLIC_IP 
4. Update the server: sudo apt update && sudo apt upgrade -y

Lab Task 2: Server Investigation and Deployment Report  
Perform a thorough investigation of your running EC2 server and write a deployment report. The 
report must be committed to your GitHub repository as deployment_report.md. 
1. From the EC2 instance, run and record the output of: docker ps, docker logs flask-api, ss -
tulnp | grep :5000, df -h, free -h, uptime 
2. From your LOCAL machine, run: curl -i http://YOUR_PUBLIC_IP:5000/api/students and 
record the full HTTP response including headers and status code 
3. Stop and restart your container: docker stop flask-api then docker start flask-api,  verify it 
recovers and is reachable again 
4. Test the --restart=always flag: reboot the EC2 instance (sudo reboot), wait 60 seconds, 
SSH back in, and verify the container started automatically without any manual intervention 
5. Write deployment_report.md with: the EC2 instance public IP, instance type, region, Docker 
version, all curl test results, and a paragraph in your own words describing what happens 
step-by-step when someone accesses your API from the internet 
6. Commit and push deployment_report.md to your GitHub repository 
7. Show the instructor the GitHub repository with the report and the live API accessible from 
the internet simultaneously 
 
 
 
 
 
   

============================================================
SUPER QUICK TROUBLESHOOTING GUIDE
============================================================
FLASK NOT OPENING ON PUBLIC IP
1. Check container/process:
docker ps
ps aux | grep app.py

2. Check port:
ss -tulnp | grep :5000

3. Check local response on EC2:
curl http://localhost:5000/api/health

4. Check public response:
curl http://YOUR_PUBLIC_IP:5000/api/health

5. Check app.py:
app.run(host='0.0.0.0', port=5000)

6. Check AWS Security Group:
Custom TCP 5000 from 0.0.0.0/0

DOCKER FIX: CONTAINER NAME ALREADY EXISTS
docker stop flask-api
docker rm flask-api
docker run -d -p 5000:5000 --restart=always --name flask-api flask-api:v1

DOCKER FIX: PERMISSION DENIED
sudo usermod -aG docker $USER
newgrp docker

POST REQUEST FIX
Use Content-Type header and valid JSON:
curl -i -X POST http://localhost:5000/api/students -H 'Content-Type: application/json' -d '{"name":"Zara","grade":"A"}'

GITHUB PUSH PASSWORD FIX
Use a GitHub Personal Access Token as password for HTTPS push. Normal GitHub password usually will not work.

STOP AWS CHARGES
EC2 -> Instances -> select instance -> Instance state -> Stop
